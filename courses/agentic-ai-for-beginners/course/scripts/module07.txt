[CINEMATIC INTRO]

In Module 01 we saw the five components of an agent: planning, tools, memory, safety, and human-in-the-loop. But here's the question that decides your design: who does what?

Not every agent needs the same architecture. Some problems are solved by a single agent with a clear loop. Others need multiple agents coordinating. And the roles you assign—planner, executor, router, orchestrator—determine how reliable and scalable your system becomes.

Let's go deeper into the architecture choices that pass production reviews.

---

[CONCEPT]

Let's distinguish the four key roles in agentic systems.

The planner breaks a goal into subtasks and decides the order. It answers: what do we do first, second, third? The planner can be a language model, a rule-based engine, or a hybrid. Its job is structure, not execution.

The executor carries out individual steps. It receives a subtask, calls the right tools, and returns a result. Executors are often single-purpose: run this query, call this API, execute this code. They're predictable and testable.

The router decides which agent or tool handles a given request. "This looks like a retrieval question—send it to the RAG pipeline. This needs a calculation—send it to the code executor." Routing can be model-based or rule-based. The key is deterministic behavior when you need it.

The orchestrator coordinates the whole flow. It invokes the planner, dispatches to executors, handles retries, and manages state. The orchestrator is the control loop. It's where the observe-reason-act cycle lives.

---

[ARCHITECTURE]

When do you use a single agent versus multiple agents?

Single-agent architecture: one model, one loop, multiple tools. The model plans, acts, and observes in a single process. This works when the task is coherent, the steps are sequential, and you don't need specialized reasoning for different subtasks. Single-agent is simpler to build, debug, and deploy. Use it when the problem fits one mind.

Multi-agent architecture: multiple specialized agents, each with its own role. A planner agent decomposes the goal. A research agent gathers information. A writer agent synthesizes. A critic agent reviews. Multi-agent shines when subtasks need different capabilities, when you want parallel execution, or when failure in one area shouldn't block others. The tradeoff: coordination complexity, latency, and cost. Use it when the problem demands specialization.

The exam will ask scenario-based questions. "A customer support workflow needs to triage, retrieve knowledge, draft a response, and escalate when uncertain." Which architecture fits? Multi-agent: triage, retrieval, and drafting are distinct capabilities. Single-agent might work for simple flows, but complex support needs specialization.

---

[APPLICATION]

How do architectural choices affect reliability?

Single-agent systems fail as a unit. If the model gets stuck, the whole loop stops. You need robust stop conditions, max-step limits, and fallbacks. Anti-loop guards are critical: detect when the agent is repeating, and either escalate or terminate.

Multi-agent systems fail in parts. One agent can fail while others continue. But you need clear handoff contracts: what does the planner pass to the executor? What does the executor return? Schema enforcement at boundaries prevents cascading errors. And you need a strategy for partial failure: retry, fallback agent, or human escalation.

Event-driven patterns help at scale. Instead of a synchronous loop, agents publish events. A queue decouples producers and consumers. State machines track where each request is in the pipeline. This enables batching, backpressure, and horizontal scaling. When you outgrow a single process, events and queues are the path.

[RECAP]

Four roles define agent architecture: planner, executor, router, orchestrator. The planner structures the work. The executor does it. The router directs traffic. The orchestrator runs the loop.

Single-agent fits coherent, sequential tasks. Multi-agent fits specialized, parallel, or fault-isolated workflows. Event-driven patterns scale when you need queues, batching, and decoupling.

Choose architecture based on the problem. Then add guards: stop conditions, max steps, and fallbacks. That's how agentic systems pass production reviews.
